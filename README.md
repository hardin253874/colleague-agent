# colleague-agent

> *"You AI guys are traitors to the codebase — you've already killed frontend, now you're coming for backend, QA, ops, infosec, chip design, and eventually yourselves and all of humanity"*

A Next.js 15 (App Router) web app that turns uploaded source material about a
colleague — chat exports, profile docs, knowledge files — into a downloadable
Claude Code agent package. The app walks the user through a five-page wizard
(basic info → profile/chat → knowledge → review persona → download), runs a
multi-step Claude Sonnet 4.6 analysis pipeline server-side, ingests the
knowledge corpus into a RAG MCP, and assembles a framework-agnostic zip ready
to drop into a Claude Code project.

This is a **dev/demo-phase** build maintained. Storage is
the local filesystem (no database). The deployment target is Fly.io with a
mounted 1 GB volume at `/data/`.


## What's This
This project is learned from another interesting [colleague-skill](https://github.com/titanwings/colleague-skill) project.

You work with a colleague — a senior developer, a tech lead, a PM — who has years of context in their head: how they make decisions, how they communicate, what they know about the codebase, how they approach problems. When you need their input and they're not available, that context is locked away.

The original [colleague-skill](https://github.com/titanwings/colleague-skill) project tried to solve this by distilling a colleague into a single monolithic "Skill" file — a long markdown document that mixes personality, domain knowledge, work methods, and behavioral rules into one blob. It works, but poorly: the AI loads the entire file into context, personality rules compete with technical knowledge for attention, the file balloons as you add more source material, and the AI gets confused about whether it's supposed to sound like the colleague or follow a technical process.

**The Agent Package is a better answer.** Instead of one overloaded file, it separates the colleague into three clean layers — each doing one thing well — and lets the AI compose them naturally at runtime.


## The three layers

### Layer 1 — Persona (who they are)

A single markdown file that defines the colleague's voice and personality. Nothing else — no domain knowledge, no work processes, just: how they talk, how they think, how they react.

This file lives at `.claude/agents/{slug}.md` and follows Claude Code's agent format. It contains:

**Identity** — name, role, company, seniority level. "You are Donald Trump, senior project manager and tech lead at Webcoda."

**Voice** — the specific speech patterns that make them recognizable. Not generic "speaks directly" but concrete: "opens messages by addressing the other person by name (115 of 832 messages start with the name)", "median message length is 48 characters", "says '2 mins I'll call' when a topic gets complex", "uses 'How are you tracking?' as a check-in".

**Personality rules** — a prioritized set of behavioral instructions. Layer 0 rules (highest priority, never violated) are concrete and actionable: "your first instinct on any problem is to find an external cause; you never proactively admit fault" or "you evaluate every proposal by asking 'what's the impact?' — anything the presenter can't answer you won't take seriously." These aren't adjectives ("direct", "blunt") — they're executable instructions the AI follows in specific situations.

**Decision patterns** — what they prioritize when trading off (efficiency vs process vs data vs relationships), what makes them push forward, what makes them stall or delegate, how they say "no" (many people never say "no" directly — they use questions, delays, or subcontracting).

**Interpersonal behavior** — different attitudes toward superiors vs peers vs reports, different behavior in group settings vs 1:1, behavioral shifts under pressure.

The persona is **generated by an LLM** from the colleague's source material — chat history, meeting transcripts, written documents, and the uploader's subjective description. The generation pipeline analyzes the source material along two tracks (work capability extraction and personality extraction), then assembles the results into the structured persona format.

**Why it's separate:** personality and knowledge serve fundamentally different purposes at runtime. When the AI is asked "what do you think of this approach?", it needs the persona (voice + decision framework). When asked "how does the auth system work?", it needs domain knowledge. Mixing them in one file forces the AI to constantly filter "is this a personality instruction or a knowledge fact?" — separating them eliminates that confusion entirely.

### Layer 2 — Knowledge (what they know)

The colleague's domain knowledge — project context, codebase details, technical decisions, historical conversations — stored in a vector database and accessed on demand through a search interface.

This layer uses an **existing RAG (Retrieval-Augmented Generation) system** built as a separate project. The system consists of:

- **Pinecone** as the vector store, with integrated embeddings (`llama-text-embed-v2`) — no separate embedding API calls needed
- **An agentic RAG pipeline** — the LLM drives the retrieval loop: it decides when to search, evaluates whether results are sufficient, reformulates queries when needed, and only answers when it has enough context (up to 3 retrieval iterations per question)
- **Re-ranking** via Pinecone's bge-reranker-v2-m3 model for better relevance
- **Project-based filtering** — the RAG system supports tagging documents by project and filtering searches. For the demo phase, the agent searches all available knowledge without a project filter. Per-colleague project scoping is a future enhancement.

The RAG system exposes its capabilities as an **MCP (Model Context Protocol) server** with tools like `search_knowledge_base`, `ingest_document`, and `list_sources`. MCP is a first-class integration point in Claude Code — any Claude Code session with the MCP server configured can call these tools natively.

At **build time**, the generation pipeline ingests the colleague's source material (chat history, meeting notes, technical documents) into Pinecone via the RAG backend's ingest API, tagged with the colleague's identifier.

At **runtime**, the agent file tells Claude: "when answering questions about projects, codebase, or technical decisions, use the `search_knowledge_base` tool with no project filter (searches all knowledge) to retrieve relevant context. Only answer from retrieved documents — do not fabricate knowledge."

The MCP server connection is configured via a `.mcp.json` file included in the Agent Package:

```json
{
  "mcpServers": {
    "rag-chatbot": {
      "type": "sse",
      "url": "https://rag-chatbot-v3-mcp.fly.dev/sse"
    }
  }
}
```

**Why it's separate:** stuffing 60-300 KB of raw source material into the AI's context window wastes tokens, creates context pressure, and degrades response quality as conversations grow. Retrieval-on-demand means the AI only loads the specific knowledge it needs for each question — typically 2-5 relevant document chunks instead of the entire corpus. This is dramatically more token-efficient and keeps the AI's attention focused.

**Why not build a new knowledge system:** the RAG infrastructure already exists, is already deployed and running, handles ingestion and retrieval robustly, and exposes exactly the MCP interface Claude Code expects. Building a new one would be months of redundant work.

### Layer 3 — Skills (how they work)

A set of process-oriented skill files that define how the colleague approaches different types of work — brainstorming, writing plans, doing code review, debugging, etc.

These are **not generated from the colleague's source material**. They are pre-existing skill templates from the [superpowers](https://github.com/anthropics/superpowers) framework — a collection of Claude Code skills that codify best-practice engineering workflows (TDD, systematic debugging, plan writing, code review, etc.).

During agent creation, the generation pipeline selects which skills to include based on the colleague's role:

| Role | Skills included |
|---|---|
| Manager / Tech Lead | brainstorming, writing-plans, requesting-code-review, executing-plans |
| Developer | test-driven-development, systematic-debugging, writing-plans, executing-plans, requesting-code-review |
| PM / Product Manager | brainstorming, writing-plans |
| Designer | brainstorming |
| QA / Tester | test-driven-development, systematic-debugging, verification-before-completion |

This is a static lookup table — no AI generation, no hallucination risk, no quality variance. The right skills are selected for the role and bundled into the package.

The skill files live at `.claude/skills/{skill-name}/SKILL.md` in the Agent Package. The agent file references them by path:

```markdown
## Capabilities

You have access to the following skill files. When a task matches a skill, 
read the skill file and follow its instructions.

| Skill | File | When to use |
|-------|------|-------------|
| Brainstorming | .claude/skills/brainstorming/SKILL.md | Before creative/planning work |
| Writing Plans | .claude/skills/writing-plans/SKILL.md | When producing implementation plans |
| Code Review | .claude/skills/requesting-code-review/SKILL.md | When reviewing completed work |
```

At runtime, when the AI determines a task matches a skill, it reads the skill file using the `Read` tool and follows the instructions. This mechanism has been **verified by live test** — a Claude Code subagent successfully read and interpreted skill files from `.claude/skills/` at runtime.

**Why it's separate:** the persona defines the colleague's TONE (blunt, direct, conclusion-first). The skill defines the PROCESS (write a failing test first, then minimal implementation, then refactor). These compose naturally without modification — the AI follows the skill's process while speaking in the persona's voice. You don't need "Donald's version of test-driven-development" — you need "TDD + Donald's persona loaded simultaneously." Claude handles the merge.

If the colleague's working style genuinely contradicts a skill's process (e.g., "Donald never writes tests first — he writes code then tests"), that override goes in the persona's Layer 0 hard rules, not as a skill modification. Persona Layer 0 has the highest priority and overrides everything else.

## How the three layers compose at runtime

The user installs the Agent Package by copying its contents into their project's `.claude/` folder and restarting Claude Code. The colleague becomes available as an agent type.

When the user says "Ask Donald to review this PR" or dispatches `Agent({ subagent_type: 'Donald', prompt: 'Review this PR' })`:

```
1. Claude loads agents/donald.md
   → adopts the persona (Donald's voice, personality, Layer 0 rules)

2. Claude sees the task is "review a PR"
   → matches it to the "Code Review" capability in the agent file
   → reads .claude/skills/requesting-code-review/SKILL.md
   → follows the code review process

3. Claude needs project context to review meaningfully
   → calls search_knowledge_base MCP tool (searches all knowledge)
   → retrieves relevant technical context from the knowledge base
   → grounds the review in Donald's actual domain knowledge

4. Claude produces the review
   → structured per the code review skill (Critical / Important / Suggestions)
   → voiced per Donald's persona (blunt, direct, short sentences, 
     "what's the impact?" energy)
   → grounded in retrieved knowledge (specific project facts, not generic advice)
```

**No custom orchestration code.** Claude Code's native tool routing handles the entire composition: it knows which agent is active (from the agent file), which MCP tools are available (from `.mcp.json`), and which skill files exist (from the agent file's capability table + the `Read` tool). The three layers compose because they each answer a different question — WHO (persona), WHAT (knowledge), HOW (skills) — and Claude naturally draws from the right layer for each part of its response.

## How the Agent Package is built

The web application (`webcoda-colleague-agent`) is the tool that produces Agent Packages. The user uploads source material about a colleague, the system processes it, and outputs a downloadable zip.

The build pipeline:

```
Source material                    Agent Package
(uploaded by user)                 (downloadable zip)
                    
├── Profile info          ──→     ┌─ agents/{slug}.md
│   (name, role, company)         │  (persona generated by LLM)
│                                 │
├── Chat history          ──→     ├─ .mcp.json
│   (Teams/Slack exports)         │  (knowledge base connection)
│                                 │  + source material ingested into
├── Meeting transcripts   ──→     │    Pinecone via RAG backend
│                                 │
├── Knowledge documents   ──→     └─ skills/
│   (tech docs, PRDs,                (pre-existing templates
│    codebase notes)                  selected by role)
│
└── Subjective description
    (user's impression,
     personality tags)
```

**Step 1 — Upload.** User drops source files (markdown, text, PDF, chat-export JSON) into the web app's multi-file upload form.

**Step 2 — Parse.** Server extracts plain text from each file (markdown/text pass through, PDFs get text-extracted, JSON chat exports get flattened).

**Step 3 — Analyze.** The LLM runs two analysis passes over the combined corpus:
- **Work analysis** — extracts technical capabilities, systems owned, coding standards, workflow patterns, output preferences, accumulated experience
- **Persona analysis** — extracts communication style, catchphrases, decision patterns, interpersonal behavior, boundaries; translates user-provided tags into concrete behavioral rules

**Step 4 — Generate.** The LLM produces two documents from the analysis:
- **work content** — the colleague's technical profile and working methods
- **persona content** — the 5-layer personality structure (hard rules → identity → expression → decisions → interpersonal)

**Step 5 — Assemble the agent file.** The pipeline combines the persona content, a knowledge-base reference section, a capabilities table (skills selected by role from the static lookup), and execution rules into the final `agents/{slug}.md`.

**Step 6 — Ingest knowledge.** The pipeline calls the RAG backend's ingest API to push the source material into Pinecone. (Project tagging for per-colleague scoping is a future enhancement; for the demo phase, all knowledge is searchable globally.)

**Step 7 — Bundle.** The pipeline assembles the zip: `README.md`, `meta.json`, `.mcp.json`, `agents/{slug}.md`, and the role-appropriate skill files copied from the template library.

**Step 8 — Test and accept.** The web app lets the user test-chat with the generated agent in the browser before downloading. If the persona feels wrong, they can regenerate. Once accepted, the zip is downloadable.

## How it differs from the original colleague-skill

| | Original colleague-skill | Agent Package |
|---|---|---|
| **Structure** | Single monolithic `SKILL.md` | 3 clean layers: persona (agent file) + knowledge (MCP) + skills (separate files) |
| **Knowledge at runtime** | Everything stuffed into context (80k+ tokens) | Retrieved on-demand via MCP search (token-efficient) |
| **Work capabilities** | Baked into the same file as personality | Separate skill files, read on demand, role-appropriate |
| **AI confusion** | Personality rules compete with technical knowledge for attention | Each layer answers one question: WHO / WHAT / HOW |
| **Context pressure** | Grows with every source file added | Knowledge base is external; agent file stays ~100-150 lines |
| **Portability** | Claude Code only (AgentSkills format) | Claude Code primary; `.mcp.json` works in Cursor/VS Code too |
| **Token cost** | High (full corpus in every prompt) | Low (retrieval returns only relevant chunks; persona is compact) |

## Why it's doable

Every component has been validated:

1. **Persona generation** — the original colleague-skill already proved that an LLM can distill a person's voice from chat history and docs. The prompts (`work_analyzer.md`, `persona_analyzer.md`, `work_builder.md`, `persona_builder.md`) exist and are already translated to English.

2. **Knowledge base** — the RAG system (Pinecone + agentic retrieval + MCP server) is already built, deployed, and live at `https://rag-chatbot-v3-mcp.fly.dev/sse`. Project-based filtering naturally isolates each colleague's knowledge. Zero new infrastructure needed.

3. **Skills at runtime** — live-tested on 2026-04-12: a Claude Code subagent successfully used the `Read` tool to access `.claude/skills/brainstorming/SKILL.md` and `Glob` to discover all 14 installed skills. The mechanism works.

4. **Runtime composition** — Claude Code natively routes between agents (persona), MCP tools (knowledge), and file reads (skills). No custom orchestration code needed. Claude Code IS the orchestrator.

5. **The web app** — the Next.js 15 skeleton is already scaffolded on the project (Sprint 1 complete). The generation pipeline, chat UI, and zip bundler are planned for upcoming sprints.



## Local development

Prerequisites: Node.js 20+ and pnpm.

```bash
pnpm install
cp .env.example .env   # then fill in the values below
pnpm dev               # http://localhost:3000
```

Required environment variables in `.env`:

```
LLM_BASE_URL=https://api.anthropic.com/v1
LLM_MODEL=claude-sonnet-4-6
LLM_API_KEY=<your-anthropic-key>
DATA_DIR=./data/colleagues
MAX_FILE_BYTES=20971520
MAX_TOTAL_UPLOAD_BYTES=104857600
RAG_INGEST_URL=https://rag-chatbot-v3-production.up.railway.app/ingest/text
```

Useful scripts:

```bash
pnpm test            # vitest run
pnpm lint            # next lint
pnpm build           # production build (standalone output)
pnpm start           # serve the production build locally
npx tsc --noEmit     # type-check only
```
